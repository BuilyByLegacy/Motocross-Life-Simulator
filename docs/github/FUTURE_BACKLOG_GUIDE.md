# Future Backlog Guide

This guide preserves long-term design ambition while protecting the App Store v1.0 launch scope.

## Launch-scope rule

Create real GitHub Issues now only from:

```text
docs/github/issues/00_CREATE_NOW_V1_LAUNCH_BLOCKERS.md
```

All other issue draft files are design backlog unless explicitly promoted by a roadmap decision. Backlog ideas should remain documented, searchable, and available for later planning, but they are not App Store v1.0 blockers.

## Important future systems that are not v1.0 launch blockers

The following systems are valuable to the long-term vision, but they should not block App Store v1.0 unless a specific, minimal dependency is promoted into the launch-blocker batch:

- Full documentary engine
- Full friends/leaderboards
- Deep AI family simulation
- Full dynamic world simulation
- Era selection
- Advanced weather
- Deep travel/camping
- Full garage museum
- Advanced social systems
- Full parent campaign
- Full pro career
- Paid expansions
- Complete sport history engine

## Promotion criteria

A future/backlog item should be promoted into v1.0 only if it is required to prove the first playable loop or pass release readiness. Promotion should include:

1. A short reason it blocks v1.0.
2. A narrowed v1.0 version of the feature.
3. Acceptance criteria that can be tested before TestFlight.
4. A link back to the broader future design so the full vision is preserved.

## Backlog handling

- Do not delete existing draft files.
- Do not import the whole `docs/github/issues/` folder when creating real issues for the current launch push.
- Keep long-term systems in design backlog, future milestones, or post-launch project views.
- Use labels such as `future`, `post-launch`, or `backlog` when these ideas are eventually imported.
